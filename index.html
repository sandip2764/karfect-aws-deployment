Karfect — AWS Cloud Deployment

A full-stack PHP/Laravel web application deployed on AWS with auto scaling, CDN delivery, private networking, and centralized monitoring — built to production standards.

EC2 ASG ALB RDS MySQL S3 CloudFront VPC NAT Gateway Internet Gateway CloudWatch SNS ACM (SSL) IAM

Architecture overview

The deployment separates compute, database, and static assets across distinct AWS services — each in its correct network tier. CloudFront sits at the edge with two origins: S3 for static files and ALB for dynamic PHP traffic.

How I built this — step by step

I started from a single server and progressively layered in AWS services until the full production architecture was running. Here is the exact sequence of decisions and actions.

01
Set up a base EC2 instance and install the full stack
Launched a public EC2 instance and manually installed everything the Laravel app needs: Nginx as the web server, PHP-FPM for processing PHP, MySQL client for database access, and Composer for dependency management. Deployed the Karfect codebase to /var/www/karfect, configured the Nginx server block in sites-available, and enabled it via symlink to sites-enabled.
02
Designed the VPC — 2 public + 2 private subnets across 2 AZs
Created a custom VPC with proper subnet separation. Public subnets host internet-facing resources (ALB, Bastion, NAT Gateway). Private subnets host the compute (EC2/ASG) and database (RDS) — completely isolated from direct internet access. Configured route tables: public subnet routes 0.0.0.0/0 through the Internet Gateway, private subnet routes 0.0.0.0/0 through the NAT Gateway for outbound-only access.
03
Provisioned RDS in the private subnet and imported the database
Launched a MySQL RDS instance inside the private subnet with no public accessibility. Created a dedicated Security Group allowing inbound port 3306 only from the EC2 Security Group — nothing else. Connected to the RDS endpoint via the bastion host, created the Karfect database, and imported the full schema and data. Updated the Laravel .env DB config to point to the RDS endpoint.
04
Moved static assets to S3 and set up CloudFront with dual origins
Created an S3 bucket, uploaded all images and static files from the Karfect app, and updated the app to reference S3/CloudFront URLs. Created a CloudFront distribution with two origins: S3 as Origin 1 (for static files — delivered from edge locations worldwide) and the ALB as Origin 2 (for dynamic PHP requests). Cache behavior rules route /images/* and /static/* to S3, everything else to ALB.
05
Verified the full app was working, then created an AMI
Tested every part of the application end-to-end — web pages, database reads and writes, image delivery via CloudFront. Once everything was confirmed working on the base EC2, created a custom AMI (Amazon Machine Image) from that instance. This AMI becomes the immutable base for all future EC2 instances in the Auto Scaling Group — it captures the exact server configuration.
06
Built Auto Scaling Group from the AMI via Launch Template
Created a Launch Template using the AMI, specifying instance type, Security Group, key pair, and IAM role. Configured the Auto Scaling Group with min: 1, desired: 2, max: 4. Set up a target tracking scaling policy based on CPU utilization — ASG automatically adds instances under load and terminates them when traffic drops. Attached the ASG to the ALB Target Group so new instances are automatically registered and health-checked.
07
Configured the Application Load Balancer with SSL
Created an ALB in the public subnets, listening on ports 80 and 443. Attached an ACM (AWS Certificate Manager) SSL certificate to the HTTPS listener. Added a redirect rule: HTTP (port 80) → HTTPS (port 443). The ALB distributes incoming requests across all healthy EC2 instances in the Target Group and performs health checks to route only to running instances.
08
Installed CloudWatch Agent and configured alarms
Installed and configured the CloudWatch Agent on EC2 instances to stream Nginx access logs, Nginx error logs, and system logs to CloudWatch Log Groups — accessible remotely without needing to SSH. Created metric alarms: CPU utilization above 70% triggers an SNS notification via email, and memory utilization is also monitored. This gives full observability without touching the server.

Security design

Every resource is restricted to the minimum access it requires. No component is publicly reachable unless it absolutely needs to be.

Resource Protocol · Port Source Reason
ALB HTTP 80HTTPS 443 0.0.0.0/0 Public-facing load balancer
EC2 (via ASG) HTTP 80 ALB SG only No direct public access to EC2
EC2 SSH SSH 22 Bastion SG only SSH restricted to bastion host
Bastion Host SSH 22 Admin IP only Single controlled entry point
RDS MySQL MySQL 3306 EC2 SG only No public access, private subnet
S3 Bucket HTTPS CloudFront OAI Direct S3 access blocked

Key learnings

  • Moving from a monolithic single-server setup to a properly layered architecture requires thinking about network tiers first — what is public, what is private, and why.
  • NAT Gateway is essential for private EC2 instances to reach the internet for outbound tasks (package installs, CloudWatch log delivery) without being reachable from the internet themselves.
  • CloudFront with two origins is the right pattern when a PHP app has both dynamic routes and heavy static assets — it separates caching strategies per origin.
  • Building an AMI from a working instance before creating the ASG ensures every auto-scaled instance starts in a known, tested state — avoiding configuration drift.
  • Security groups work best when chained: ALB SG → EC2 SG → RDS SG, so no layer is directly exposed beyond its upstream caller.
  • CloudWatch Agent enables remote log access and alerting without SSH — critical for production environments where direct server access should be rare.
